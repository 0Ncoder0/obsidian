# 技术架构与模块职责

## 1. 总体架构

- 前端单体应用（Vite + TypeScript）。
- `GameApp` 作为编排入口，驱动 Physics/Audio/UI/Persistence 子系统。
- 主循环负责更新顺序与模式状态同步。

## 2. 模块分层

- `app/`：应用状态编排、模式切换、输入路由。
- `scene/`：实体数据模型、编辑操作、事件记录。
- `physics/`：Matter.js 封装、预测计算、相机跟随控制。
- `audio/`：音频合成与可视化渲染。
- `ui/`：面板、HUD、Timeline、状态提示。
- `persistence/`：序列化与本地自动保存。

## 3. 关键类职责

- `GameApp`：初始化系统、驱动主循环、处理跨系统事件。
- `ModeController`：管理编辑/播放状态机和权限闸门。
- `InputController`：统一键鼠输入并按模式分发。
- `PhysicsWorld`：实体与刚体映射、步进模拟、碰撞事件输出。
- `PredictionEngine`：基于场景快照生成编辑态预测轨迹。
- `CameraFollowController`：管理自动跟随与手动相机协调。
- `AudioEngine`：处理触发、voice 管理、混音输出。
- `TimelineRecorder`：记录 `ContactEvent` 序列。
- `LocalSaveRepository`：节流保存和恢复。

## 4. 主循环建议顺序

1. 拉取输入并执行可用操作（受模式限制）。
2. 更新模式相关 UI 状态。
3. 播放态下推进物理步进并收集碰撞触发。
4. 处理音频触发与可视化数据更新。
5. 更新相机（自动跟随或手动控制）。
6. 渲染画布和 HUD。
7. 触发自动保存（节流条件满足时）。

## 5. 状态机与门禁

- 两态：`edit`、`play`。
- 从 `edit -> play`：
  - 锁编辑。
  - 隐藏右侧面板与 Timeline。
  - 判断是否启用自动跟随（依据选中球）。
- 从 `play -> edit`：
  - 停止模拟。
  - 恢复编辑权限并显示编辑 UI。
  - 展示本轮 Timeline 结果。

## 6. 性能与稳定性约束

- 固定步进 + 渲染分离，减少帧率波动影响。
- 预测线仅编辑态计算与渲染，降低播放态负载。
- Timeline 播放态只记录不渲染，减少 UI 更新开销。
- 存档节流避免高频写入主线程阻塞。

## 7. 可观测性建议

- 关键日志点：模式切换、存档失败、音频上下文异常、碰撞触发数量。
- 开发诊断开关：显示 FPS、活跃 voice 数、每秒事件量（仅开发模式）。
