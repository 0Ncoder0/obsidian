# 01 -- 产品愿景与范围定义

## 1. 产品定位

**Marble Music** 是一款基于浏览器的 2D 物理音乐沙盒。玩家通过摆放积木搭建结构，释放弹珠，让弹珠在重力和碰撞的驱动下滚动弹跳，每次撞击音乐方块都会触发一个音符。玩家的目标不是"赢"，而是通过反复实验，搭建出有趣的声音序列。

**核心体验**：搭建 -> 聆听 -> 理解 -> 调整 -> 再来一次。

---

## 2. 第一版目标

| 目标           | 具体描述                                                                 |
| -------------- | ------------------------------------------------------------------------ |
| 闭环可用       | 完成"编辑搭建 -> 播放触发 -> 停止回看 -> 调整再来"的完整循环             |
| 可解释性       | 通过预测线（编辑态轨迹预览）和 Timeline（事件回看）让结果可预判、可复盘   |
| 低上手门槛     | 仅三种积木（小球、方块、音乐方块），无需音乐背景即可上手                  |
| 即时反馈       | 碰撞到音乐方块即刻发声，物理行为实时可见                                  |

---

## 3. 目标用户画像

**主要用户**：喜欢"实验型创作"的轻度休闲玩家

- 对音乐和物理有好奇心，但不需要专业作曲或编程背景
- 习惯 PC 键鼠交互
- 可接受浏览器内离线游玩（无需安装）
- 寻求短时间内获得创作满足感的体验（10-30 分钟一个 session）

**典型使用场景**：

- "我想搭一个斜坡让弹珠滚下去敲出一段旋律"
- "我想看看两个弹珠同时弹跳会碰撞出什么声音"
- "我想调整方块位置，让弹珠走不同的路径，听听效果有什么不同"

---

## 4. 核心体验支柱

| 支柱         | 描述                                                               |
| ------------ | ------------------------------------------------------------------ |
| 沙盒自由     | 玩家自主布置空间结构和触发路径，没有"正确答案"                     |
| 物理因果     | 小球运动完全由物理引擎驱动，结果由布局决定，不靠脚本预设           |
| 音乐反馈     | 碰撞即发声，参数可调（音高、音量、音长），听感可控                 |
| 时序回看     | 通过 Timeline 事件记录理解"为什么会这样"，支撑迭代调整             |

---

## 5. 第一版范围（In Scope）

### 实体系统

- **小球（Ball）**：动态物理实体，受重力影响。场景支持多个小球。
- **方块（Block）**：静态碰撞实体，用于搭建轨道、边界、结构。
- **音乐方块（Music Block）**：静态触发实体，被小球碰撞后触发音符。

### 音乐系统

- 单一音色：钢琴（`piano`）
- 每个音乐方块可独立配置：音名（`noteName`）、音量（`volume`）、音长（`durationMs`）
- 触发规则：每次碰撞创建独立音符实例，完整播放包络，允许重叠发声

### 模式系统

- 两种模式：**编辑**（搭建、配置）和 **播放**（物理模拟、发声触发）
- 模式强隔离：播放态禁止一切编辑操作
- `Space` 切换：编辑 -> 播放 / 播放 -> 编辑，不提供暂停

### 辅助可视化

- **预测线**：编辑态显示小球可能的运动轨迹预览
- **Timeline**：编辑态展示上一轮播放中的碰撞事件记录
- **音乐活动可视化**：发声时的脉冲/能量视觉反馈

### 相机与跟随

- 多球时可选中某个小球
- 按 `Space` 启动播放：若有选中球则自动跟随，否则不自动跟随
- 用户始终可手动平移（中键拖拽 / `Alt+左键`）和缩放（滚轮）

### 持久化

- 浏览器 `localStorage` 自动保存
- 刷新页面后恢复场景结构和音乐参数

## 6. 第一版明确不做（Out of Scope）

| 排除项               | 原因                                         |
| -------------------- | -------------------------------------------- |
| MusicXML 导入        | v1 聚焦沙盒创作，不需要外部乐谱              |
| 场景/乐谱导入导出    | 涉及文件格式设计和兼容性，推迟到后续版本      |
| 网络协作 / 账号 / 云存档 | 需要后端基础设施，超出 v1 技术范围         |
| 移动端适配           | 交互模型基于键鼠，触屏需要重新设计            |
| 多音色 / 音源管理    | v1 固定钢琴音色，降低复杂度                   |
| 暂停功能             | 设计决策：简化模式状态机，仅播放/停止         |

---

## 7. 成功标准

### 功能验收

| 编号 | 标准                                                                       |
| ---- | -------------------------------------------------------------------------- |
| S1   | 玩家可在 10 分钟内独立完成一次"搭建并触发声音"的完整闭环                   |
| S2   | 播放模式下不存在任何可触发编辑的路径（新增、移动、删除、参数修改均被阻断） |
| S3   | 刷新页面后场景结构与音乐参数完整恢复                                        |
| S4   | 玩家能通过 Timeline 理解碰撞触发的时间先后关系                              |

### 非功能要求

| 编号 | 标准                                                                       |
| ---- | -------------------------------------------------------------------------- |
| N1   | 标准场景（<= 20 积木、<= 5 小球）下体感流畅，无明显掉帧                   |
| N2   | 模式切换（编辑 <-> 播放）无可感知卡顿或状态错乱                            |
| N3   | 存档失败时用户可感知（有明确提示，不静默丢失数据）                          |
