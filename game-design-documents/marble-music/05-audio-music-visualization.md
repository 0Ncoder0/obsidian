# 05 -- 音频与音乐可视化设计

## 1. 音频设计目标

| 目标             | 说明                                                       |
| ---------------- | ---------------------------------------------------------- |
| 低延迟触发       | 碰撞发生到声音可闻的延迟应低于 20ms，确保"即时反馈"感      |
| 音色单一且可控   | v1 固定钢琴音色，降低复杂度，聚焦核心体验                  |
| 音量驱动衰减     | 声音长度由初始音量自然决定，无显式音长参数                  |
| 多声部重叠       | 允许多个音符同时发声，形成和弦或快速音序效果               |

---

## 2. 音乐参数语义

每个音乐方块携带以下参数，在碰撞触发时传递给音频系统：

| 参数          | 类型   | 范围/约束     | 语义                                           |
| ------------- | ------ | ------------- | ---------------------------------------------- |
| `noteName`    | string | 如 `C4`, `A3` | 音名，决定发声音高（映射到频率）               |
| `volume`      | number | 0 ~ 1         | 触发瞬间的初始音量；同时决定自然衰减时长       |
| `timbre`      | string | 固定 `piano`  | v1 锁定钢琴音色，不暴露配置                    |

**无 `durationMs` 参数**：声音的持续时长完全由 `volume` 驱动的自然衰减决定。音量越高，衰减时间越长（声音越持久）；音量越低，衰减时间越短（声音很快消失）。这模拟了真实钢琴的物理特性——用力击键产生更响且更持久的音符。

### 音名到频率的映射

使用标准十二平均律。示例：

| 音名 | 频率 (Hz) |
| ---- | --------- |
| C3   | 130.81    |
| C4   | 261.63    |
| A4   | 440.00    |
| C5   | 523.25    |

## 3. 触发模型（定稿）

### 触发条件

小球（Ball）与音乐方块（Music Block）发生物理碰撞。

### 触发行为

```
碰撞事件 ──> AudioEngine 接收 ──> 创建新的 voice 实例
                                        │
                                        ├── 读取音乐方块参数
                                        ├── 设置音高（noteName → 频率）
                                        ├── 设置初始音量（volume）
                                        ├── 由 volume 计算衰减包络
                                        └── 开始发声
```

### 核心规则

| 规则                 | 说明                                                     |
| -------------------- | -------------------------------------------------------- |
| 独立实例             | 每次触发都创建一个全新的 voice 实例，独立于其他 voice     |
| 允许重叠             | 多次触发之间允许重叠发声，不互斥、不重置旧实例           |
| 无冷却抑制           | 不使用冷却时间（cooldown）策略，快速连续碰撞照常触发     |
| 音量驱动衰减         | 每个 voice 的衰减时长由初始音量决定，无显式音长参数      |

---

## 4. 衰减模型（音量驱动）

不使用传统的 ADSR 四阶段包络。声音长度完全由初始音量的自然衰减决定：

```
音量
 ▲
 │  ╲
 │   ╲
 │    ╲
 │     ╲
 │      ╲
 │       ╲___
 └──────────────► 时间
  A    自然衰减
```

### 模型说明

| 阶段       | 行为                                                           |
| ---------- | -------------------------------------------------------------- |
| Attack     | 极短（5 ~ 10 ms），从 0 快速上升到初始音量，避免点击噪声       |
| 自然衰减   | 从初始音量开始，按指数曲线平滑衰减至零                         |

### 音量与衰减时长的关系

衰减时长是初始音量的函数。建议映射方式：

```
衰减时长 = BASE_DECAY + volume * VOLUME_DECAY_SCALE
```

| 参数                 | 建议值       | 说明                                     |
| -------------------- | ------------ | ---------------------------------------- |
| `BASE_DECAY`         | 200 ms       | 最小衰减时长（即使 volume 接近 0）       |
| `VOLUME_DECAY_SCALE` | 2000 ms      | 满音量时额外增加的衰减时长               |

示例：

| 初始音量 | 衰减时长（约）| 听感                   |
| -------- | ------------- | ---------------------- |
| 0.1      | ~400 ms       | 极短的轻触音           |
| 0.5      | ~1200 ms      | 中等长度的标准音       |
| 1.0      | ~2200 ms      | 长而饱满的强音         |

### 听感要求

- 触发瞬间有明确的音头（Attack），让玩家感知到"碰撞发声"
- 音符从初始音量开始自然衰减，而非突然截断
- 衰减结束后音量归零，voice 实例可被回收
- 总体听感类似钢琴击键：轻击短促，重击绵长

## 5. 音频系统架构

### 模块职责

| 模块           | 职责                                                       |
| -------------- | ---------------------------------------------------------- |
| `PianoSynth`   | 音名 -> 频率映射，生成单次发声实例（oscillator + gain 配置）|
| `AudioEngine`  | 接收碰撞触发事件，调度 voice 创建/生命周期管理/混音输出    |

### Voice 生命周期

```
创建 ──> Attack（极短起音）──> 自然衰减（由 volume 决定时长）──> 归零 ──> 回收销毁
```

- 每个 voice 拥有独立的 OscillatorNode + GainNode
- 衰减完毕后 voice 应被清理，释放 AudioNode 资源
- 不需要对象池（v1 规模下即时创建/销毁即可）

### 限流策略

为防止极端碰撞场景（如小球卡在两个方块之间高频弹跳）导致性能问题：

| 措施                 | 说明                                                   |
| -------------------- | ------------------------------------------------------ |
| 同帧 voice 上限      | 单帧内最多创建 N 个新 voice（建议 N = 8 ~ 16）        |
| 总活跃 voice 上限    | 全局同时存在的 voice 不超过 M 个（建议 M = 32 ~ 64）  |
| 超限处理             | 超限时跳过新触发（不发声）                             |
| 日志                 | 超限时记录诊断日志，便于调试                           |

**重要**：限流不改变"可重叠"的设计原则。限流是性能保护，不是设计约束。

## 6. 音乐活动可视化

### 设计目标

让玩家在视觉上感知到"当前有声音在发生"以及"声音的强弱变化"。

### 数据输入

可视化系统从 AudioEngine 获取以下实时数据：

| 数据              | 用途                           |
| ----------------- | ------------------------------ |
| 当前活跃 voice 数 | 整体活动强度                   |
| 各 voice 当前增益 | 单个音符的实时音量             |
| 触发源位置        | 音乐方块坐标，用于定位可视化  |

### 可视化表现

| 表现形式       | 触发条件       | 行为                                          |
| -------------- | -------------- | --------------------------------------------- |
| 脉冲环         | 音乐方块发声   | 从方块位置向外扩散的环形波纹                  |
| 亮度/尺寸缩放  | 音量变化       | 音量越高，脉冲环越大越亮                      |
| 平滑衰退       | voice 衰减     | 脉冲环随音量衰减逐渐缩小并淡出               |
| 色彩映射       | 音高不同       | 可选：不同音高区间使用不同色调（暖色低音，冷色高音） |

---

## 7. 边界与异常处理

| 场景                       | 处理方式                                                   |
| -------------------------- | ---------------------------------------------------------- |
| 浏览器未授权音频上下文     | 显示提示要求用户交互以解锁音频；物理播放可继续（静音模式） |
| 高频碰撞触发               | 启用限流策略（见第 5 节），保证系统不崩溃                  |
| 音频上下文被浏览器挂起     | 下次触发时尝试恢复（resume），若恢复失败记录告警           |
| 首帧触发延迟               | 首次用户交互后显式调用 `AudioContext.resume()`             |
